# vibecleaner UI/UX 개선 계획

## 1. 목표

vibecleaner를 처음 사용하는 사람도 `이미지 추가 → 번역 → 검수 → 내보내기` 흐름을 망설임 없이 완료할 수 있게 만들고, 반복 작업을 하는 숙련 사용자에게는 빠르고 안정적인 편집 환경을 제공한다.

디자인 목표는 화려함보다 정돈된 편집 도구의 신뢰감이다. 장식은 줄이고 정보 계층, 여백, 타이포그래피, 반응의 완성도로 고급스러움을 만든다.

### 핵심 원칙

1. 다음 행동이 항상 보인다.
2. 시스템이 무엇을 하고 있는지 숨기지 않는다.
3. 기본 흐름은 단순하게, 전문 옵션은 필요할 때만 보여준다.
4. 되돌릴 수 있는 행동은 빠르게, 되돌릴 수 없는 행동은 명확하게 확인한다.
5. 고급스러움은 일관성과 절제에서 만든다.

## 2. 현재 상태 진단

### 잘 갖춰진 기반

- 페이지 목록, 캔버스, 인스펙터로 구성된 3열 편집 구조는 작업 맥락과 잘 맞는다.
- 번역 취소, 진행률, 저장되지 않은 상태, 토스트 실행 취소 등 중요한 피드백 장치가 이미 있다.
- 라이트·다크 테마, 디자인 토큰, 키보드 포커스, 모션 감소 설정이 마련되어 있다.
- 다중 페이지 선택, 페이지 검색, 드래그 재정렬, 말풍선 직접 편집 등 숙련 사용자 기능이 존재한다.

### 우선 해결할 문제

| 우선순위 | 문제 | 사용자 영향 | 현재 근거 |
| --- | --- | --- | --- |
| P0 | 이미지가 없을 때 중앙 작업 영역의 시작 행동이 약하다 | 첫 사용자가 어디서 시작할지 탐색해야 한다 | `Canvas`는 이미지 URL이 없으면 핵심 빈 상태를 렌더링하지 않는다 |
| P0 | 번역이 한 번의 큰 작업으로만 보인다 | 준비 상태, 처리 단계, 실패 위치를 이해하기 어렵다 | 주 CTA와 하단 진행률은 있으나 단계별 상태·사전 확인이 없다 |
| P0 | 설정에서 기본 사용자와 전문가가 같은 밀도로 옵션을 본다 | 용어 부담과 잘못된 설정 위험이 커진다 | 번역·감지·OCR·인페인팅 세부 옵션이 동일 계층에 노출된다 |

| P1 | 좌우 패널 폭이 고정되어 있다 | 작은 화면과 긴 파일명·번역문에서 작업 공간을 조절하기 어렵다 | `--sidebar-width`, `--inspector-width`가 고정 토큰이다 |
| P1 | 진행 피드백이 24px 상태바에 집중된다 | 장시간 처리 중 앱이 멈춘 것처럼 느낄 수 있다 | 처리 단계는 상태바와 버튼 morph에 주로 표현된다 |
| P1 | 시각 효과의 강조점이 여러 곳에서 경쟁한다 | 전문 편집 도구보다 장식적인 인상을 줄 수 있다 | 글래스, 광택 그라데이션, 글로우, 반복 shimmer가 함께 사용된다 |
| P2 | Windows 앱인데 macOS형 신호등 버튼을 사용한다 | 플랫폼 기대와 동작 의미가 어긋날 수 있다 | 커스텀 빨강·노랑·초록 창 제어 버튼을 사용한다 |
| P2 | 컴포넌트별 인라인 스타일과 개별 CSS가 혼재한다 | 미세한 간격·상태 표현이 점차 달라질 위험이 있다 | `Toolbar`, `Canvas`, 모달과 별도 CSS 파일에 스타일이 분산된다 |

## 3. 목표 사용자 흐름

```text
첫 실행
  → 언어와 품질 프리셋 선택
  → 필요한 모델 준비 상태 확인
  → 이미지 추가
  → 번역 범위와 예상 작업 확인
  → 단계별 처리 진행
  → 문제 있는 말풍선 우선 검수
  → 원본/결과 비교
  → 내보내기
```

항상 다음 세 가지가 화면에서 답을 가져야 한다.

- 지금 무엇을 선택했는가?
- 지금 앱은 무엇을 하고 있는가?
- 다음에 할 수 있는 가장 중요한 행동은 무엇인가?

## 4. 정보 구조 개선안

### 상단 바

- 좌측: 앱/프로젝트 이름과 저장 상태.
- 중앙: 현재 페이지명과 `현재 페이지 / 전체 페이지` 위치.
- 우측: `이미지 추가`, `저장`, `내보내기`, 환경설정. 저빈도 항목만 더보기 메뉴로 이동.
- Windows에서는 네이티브에 가까운 최소화·최대화·닫기 버튼을 우측에 배치한다.
- 저장 상태는 작은 주황 점만 쓰지 않고 `저장됨`, `저장 중`, `저장 필요`를 툴팁과 접근성 레이블로 제공한다.

### 왼쪽 페이지 패널

- 상단에 `페이지` 제목, 개수, 추가 버튼을 한 줄로 정돈한다.
- 선택된 페이지의 수를 고정 영역에 표시하고, 다중 선택 시 `3개 선택됨 · 번역 · 내보내기` 컨텍스트 바를 노출한다.
- 썸네일 상태를 `미처리`, `처리 중`, `검수 필요`, `완료`의 작은 상태 마커로 표시한다.
- 패널 너비를 220–360px 범위에서 조절 가능하게 하고 사용자 설정을 기억한다.

### 중앙 캔버스

- 이미지가 없을 때 중앙에 명확한 시작 카드 제공:
  - `이미지를 여기에 놓으세요`
  - `이미지 선택` 기본 버튼
  - `프로젝트 열기` 보조 버튼
  - 지원 형식과 단축키 안내
- 이미지가 있을 때 하단 플로팅 바는 `번역`, `비교`, `줌` 세 그룹으로 분리한다.
- 계속 움직이는 번역 버튼 shimmer는 제거한다. 기본 상태는 단색, 처리 중에만 절제된 진행 애니메이션을 사용한다.
- 원본/결과 비교는 `길게 눌러 원본 보기`와 분할 슬라이더 중 하나를 1차 사용자 테스트로 결정한다.
- 감지 박스 디버그 기능은 일반 툴바에서 제외하고 개발자/고급 메뉴로 이동한다.

### 오른쪽 인스펙터

- 선택 전에는 단순 안내 대신 `말풍선을 선택해 텍스트와 스타일을 편집하세요`와 주요 단축키를 보여준다.
- 선택 후 헤더에 말풍선 번호, 상태, 이전/다음 문제 이동 버튼을 둔다.
- 탭은 `텍스트`, `스타일`을 유지하되 저장 방식이 자동인지 명확히 표시한다.
- 원문과 번역문 사이에 `재인식`, `재번역`을 각각 가까이 배치하고 작업 범위를 명확히 한다.
- 문제 섹션은 페이지 아래쪽에 묻지 않고 문제가 있을 때 상단에 요약 배너를 노출한다.
- 패널 너비를 300–440px 범위에서 조절 가능하게 한다.

### 하단 상태 영역

- 유휴 상태: 현재 페이지, 줌, 선택 수, 저장 상태만 표시.
- 처리 상태: `감지 → OCR → 번역 → 정리 → 렌더링` 단계, 페이지 진행 수, 남은 작업 힌트, 취소 버튼 표시.
- 오류 상태: 실패 단계, 영향받은 페이지, `재시도`와 `자세히 보기`를 같은 위치에서 제공.
- 장시간 작업은 하단 상태바 외에 캔버스 우측 하단에 접을 수 있는 작업 패널을 제공한다.

## 5. 시각 디자인 방향: Editorial Studio

만화 원고가 주인공이고 UI는 조용한 편집 도구로 남는 방향이다.

### 색상

- 기본 표면은 `Graphite` 계열 다크와 `Porcelain` 계열 라이트를 사용한다.
- 브랜드 포인트는 현재 시스템 블루보다 채도를 약간 낮춘 코발트 하나로 제한한다.
- 성공·경고·오류 색은 상태 표현에만 사용하고 장식에는 쓰지 않는다.
- 한 화면에서 강한 그림자와 글로우는 주요 CTA 또는 떠 있는 패널 한 곳에만 허용한다.

### 표면과 깊이

- 글래스 효과는 상단 바와 플로팅 컨트롤에만 제한한다.
- 좌우 패널은 불투명 표면과 1px 구분선으로 캔버스와 분리한다.
- 카드 중첩을 줄이고 관련 항목은 여백과 얇은 구분선으로 묶는다.
- 모서리 반경은 컨트롤 8px, 카드 12px, 모달 16px의 세 단계로 축소한다.

### 타이포그래피

- UI 기본 글꼴은 Pretendard/Segoe UI 계열로 통일하고 숫자·진행률은 tabular 숫자를 사용한다.
- 기본 본문 13px, 보조 정보 11–12px, 섹션 제목 12px, 화면 제목 15–16px로 계층을 명시한다.
- 대문자 섹션 제목과 과도한 letter-spacing은 줄여 한국어에서도 자연스럽게 보이게 한다.

### 모션

- hover 120ms, 패널/모달 180–220ms, 작업 상태 전환 240ms로 제한한다.
- 반복 애니메이션은 실제 처리 중인 경우에만 사용한다.
- 성공은 짧은 체크 전환, 오류는 흔들림 대신 색·아이콘·문장으로 표현한다.

## 6. 핵심 UX 개선 상세

### A. 첫 실행과 빈 상태

1. 초기 설정을 `언어 → 품질 프리셋 → 모델 준비`의 3단계로 재구성한다.
2. `균형`, `빠름` 프리셋에 속도·품질·다운로드 크기 설명을 붙인다.
3. 기술 용어는 `고급 설정` 안으로 이동한다.
4. 완료 후 빈 캔버스에 이미지 추가 행동을 바로 연결한다.

완료 기준:

- 신규 사용자가 도움말 없이 첫 이미지를 60초 안에 추가할 수 있다.
- 초기 설정 중 현재 단계와 남은 단계가 항상 보인다.
- 건너뛰기의 결과와 나중에 설정할 위치가 명시된다.

### B. 번역 실행과 진행

1. 번역 버튼 클릭 시 범위가 애매할 때만 작은 사전 확인 팝오버를 연다.
2. 팝오버에 페이지 수, 원본/대상 언어, 품질 프리셋을 요약한다.
3. 처리 중 현재 파이프라인 단계와 페이지 진행 수를 보여준다.
4. 부분 실패 시 성공한 결과를 유지하고 실패 페이지만 재시도할 수 있게 한다.
5. 완료 후 `검수 필요 N개`를 다음 주요 행동으로 제안한다.

완료 기준:

- 처리 시작 후 500ms 안에 시각적 피드백이 나타난다.
- 사용자는 언제든 현재 단계, 처리 범위, 취소 가능 여부를 알 수 있다.
- 실패 메시지는 원인, 영향, 해결 행동을 포함한다.

### C. 검수와 편집

1. 말풍선에 `미검수`, `검수 필요`, `완료` 상태를 부여한다.
2. 저신뢰 OCR, 빈 번역, 텍스트 넘침을 검수 큐로 모은다.
3. `다음 문제` 단축키로 문제 말풍선을 순회한다.
4. 원문/번역문 편집은 자동 저장하되 저장 중·완료 상태를 작게 표시한다.
5. 삭제, 위치 변경, 텍스트 변경은 동일한 실행 취소 모델을 사용한다.

완료 기준:

- 문제가 있는 말풍선을 페이지 전체에서 직접 찾을 필요가 없다.
- 자동 저장 실패가 조용히 사라지지 않는다.
- 키보드만으로 말풍선 이동, 텍스트 편집, 다음 문제 이동이 가능하다.

### D. 설정 단순화

1. 기본 보기에는 언어, 번역 제공자, 품질 프리셋, 테마만 노출한다.
2. OCR 임계값, 마스크 확장, 생성 파라미터는 `고급` 보기로 이동한다.
3. 숫자 옵션에는 권장 범위와 `기본값 복원`을 제공한다.
4. API 연결은 저장 전에 테스트할 수 있게 하고 성공/실패를 필드 가까이 표시한다.
5. 설정 변경이 현재 작업부터 적용되는지 다음 작업부터 적용되는지 설명한다.

완료 기준:

- 기본 설정 화면에서 전문 용어가 필수 경로를 막지 않는다.
- 잘못된 키나 URL은 모달을 닫기 전에 발견할 수 있다.
- 각 고급 옵션에 기본값과 영향 설명이 있다.

## 7. 구현 로드맵

### Phase 0 — 기준선과 설계 확정 (2–3일)

- 핵심 작업 5개에 대한 휴리스틱 점검: 첫 실행, 가져오기, 단일 번역, 일괄 번역, 내보내기.
- 대표 화면 크기 1280×720, 1440×900, 1920×1080에서 기준 캡처.
- 디자인 토큰과 공통 상태 목록 확정.
- 5명 내외의 사용자에게 현재 버전으로 첫 번역 과업 테스트.

산출물: 기준 화면, 과업 성공률, 문제 목록, 확정된 화면 구조.

### Phase 1 — 발견성과 상태 피드백 (1주, P0)

- 중앙 빈 상태와 드롭존 구현.
- 상단 바에 이미지 추가·저장·내보내기 노출.
- 번역 사전 요약과 단계별 진행 표시.
- 오류 메시지에 재시도 행동 추가.
- 설정을 기본/고급으로 분리.

대상: `App.tsx`, `Toolbar.tsx`, `Canvas.tsx`, `StatusBar.tsx`, `SettingsModal.tsx`, `InitialSetupModal.tsx`.

### Phase 2 — 검수 중심 워크플로 (1–2주, P0/P1)

- 페이지와 말풍선 검수 상태 모델 추가.
- 문제 큐, 다음/이전 문제 이동, 자동 저장 상태 구현.
- 원본/결과 비교 기능 추가.
- 다중 선택 컨텍스트 바 구현.

대상: `Sidebar.tsx`, `CanvasImageStage.tsx`, `Inspector.tsx`, `InspectorProblemsSection.tsx`, 관련 hooks와 API DTO.

### Phase 3 — 디자인 시스템 정리 (1주, P1)

- `Button`, `IconButton`, `Field`, `Card`, `SegmentedControl`, `Popover`, `EmptyState`, `Progress` 공통 컴포넌트화.
- 인라인 CSS를 컴포넌트 스타일 계층으로 이동.
- 표면, 타이포그래피, 반경, 그림자, 모션 토큰 정리.
- 상시 shimmer와 과도한 글로우 제거.
- 플랫폼에 맞는 창 제어 버튼 적용.

대상: `styles/tokens.css`, `styles/apple_theme.css`, 각 컴포넌트 인라인 스타일.

### Phase 4 — 적응형 레이아웃과 접근성 (1주, P1/P2)

- 좌우 패널 리사이즈와 폭 기억.
- 좁은 화면에서 인스펙터를 오버레이/접이식 패널로 전환.
- 전체 키보드 탐색, 메뉴 roving focus, 대화상자 focus trap 점검.
- 대비, 40×40px 권장 클릭 영역, 스크린리더 레이블, 모션 감소 검증.
- 한국어·영어에서 잘림과 줄바꿈 QA.

### Phase 5 — 검증과 마감 (3–5일)

- 신규·숙련 사용자 과업 테스트 반복.
- 라이트/다크 및 세 화면 크기 시각 회귀 검사.
- 실제 1, 20, 200페이지 프로젝트 성능 확인.
- 지표와 피드백을 바탕으로 P1 잔여 항목 조정.

## 8. 성공 지표

| 지표 | 목표 |
| --- | --- |
| 첫 이미지 추가 성공률 | 95% 이상 |
| 첫 번역 완료율 | 90% 이상 |
| 첫 이미지 추가까지 걸린 중앙값 | 60초 이하 |
| 번역 실행 전 범위 오인 | 테스트 참여자의 5% 이하 |
| 처리 중 “멈춘 것 같다”는 응답 | 10% 이하 |
| 문제 말풍선 발견 시간 | 기존 대비 50% 단축 |
| 기본 설정 완료 시간 | 2분 이하 |
| 키보드만으로 핵심 편집 과업 완료 | 100% |
| WCAG AA 텍스트 대비 위반 | 0건 |

## 9. 릴리스 완료 체크리스트

- 빈 프로젝트, 단일 페이지, 다중 페이지에 각각 명확한 다음 행동이 있다.
- 번역 준비, 실행, 성공, 부분 실패, 취소 상태가 모두 설계되어 있다.
- 기본 설정과 고급 설정의 경계가 명확하다.
- 삭제 및 편집 작업의 실행 취소 정책이 일관적이다.
- 모든 아이콘 버튼에 레이블 또는 툴팁이 있다.
- 포커스 순서와 대화상자 포커스 복귀가 올바르다.
- 라이트·다크 테마에서 상태 색상이 의미를 유지한다.
- 1280×720에서도 캔버스의 주요 행동이 가려지지 않는다.
- 반복 애니메이션은 실제 처리 상태에서만 동작한다.
- 사용자 테스트와 시각 회귀 결과가 문서화되어 있다.

## 10. 가장 먼저 구현할 5개 항목

1. 이미지가 없는 캔버스의 시작 카드와 드롭존.
2. 번역 범위·언어·프리셋 요약과 단계별 진행 표시.
3. 설정의 기본/고급 분리와 연결 테스트.
4. 검수 필요 말풍선 큐와 다음 문제 이동.
5. 상시 shimmer 제거, 표면·타이포그래피·간격 토큰 정리.

이 순서는 사용성 위험을 먼저 제거하면서 시각 개편이 실제 작업 흐름 개선과 함께 진행되도록 정한 것이다.
